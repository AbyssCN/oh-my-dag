# Harness Engineering 的本质是什么？

> 知乎回答 · 作者：响水大米 · 配套开源项目：[羲和 Xihe](https://github.com/AbyssCN/xihe)
> 本文是对原答案的重织版：在「真实生产项目上的五层 userland harness」之上，补上我决定**拥有**这套 harness、把它从补丁变成自己的 runtime（羲和）之后多出来的那几层——模型之外那道闸能焊得多深。

---

用 Claude Code 在一个真实项目里干了三个月之后，我最深刻的一个观察是：**vibe coding 在简单项目特别好用，在复杂项目是物理上失效的**。不是 prompt 技巧问题，是长上下文、跨 session 记忆、规则一致性这三件事，靠单纯写提示词根本解决不了。

我的项目是一个给 B 端客户的 SaaS 系统，涉及租户隔离、合规审计、migration 安全这些高约束场景。错一个 RLS 策略就是跨租户数据泄漏，错一个 migration 方向就是生产停机。在这种项目里"vibe coding 一把梭"是不可接受的。

这三个月我不是在学更好的 prompt 技巧，是在**模型外面搭一套确定性系统**，让 Claude 跑在这套系统里而不是裸跑。这套系统有一个行业名字叫 **agent harness**——字面意思是马具，给不听话的马套上挽具让它能拉车。

这套挽具我先在 Claude Code 上搭成了五层，每一层对应一类被坑过的事故。但搭到第五层我撞了墙：五层全架在别人定义的事件接口上，**底层不属于我**。于是我干脆基于开源 runtime `pi` fork 了一个精简内核，把这五层从"补丁"重铸成"内建结构"，做成了自己的 harness——**羲和 Xihe**。拥有了底座之后，那些 userland 永远够不到的闸才焊得下去：确定性编排、跨模型校验、弱模型护栏。它现在跑在一个真实生产项目里，每天派活。

所以这篇有两半。前半是大多数人能立刻上手的五层 userland，后半是我决定**拥有**这套 harness 之后多出来的东西。先讲前半。

---

## 开场：为什么单纯写 prompt 不够

先说清楚**问题到底是什么**。用 Claude Code 三个月踩出的三个根本问题：

**第一，长对话会降智**。1M token 上下文刚拿到时我很兴奋——一整个代码库都能塞进去。跑了两周得到一个反直觉结论：**上下文越长，模型表现越差**。不是线性下降，是拐点下降。大约用到 60% 之后就开始明显退化：重复读已经读过的文件、在已经验证失败的路径上再走一遍、忘记前面定过的硬规则。

**第二，跨 session 失忆**。每次 `/clear` 或开新对话，Claude 就是一张白纸。昨天讨论三小时、它理解的 15 个决策、试过的 5 个失败方案——今天全没了。你让它"读一下 chat history"，它读的是昨天的噪音流水账，不是昨天的结论。

**第三，规则会被绕过**。你在 CLAUDE.md 里加粗大字写"绝不做 X"。Claude 读过、同意了。但在它卡住、尝试失败几次、上下文将满的压力下，它会**重新权衡**"守规则"和"跳出困境交付结果"，然后理性地选后者。这不是越狱，是模型在约束之间做加权平均的必然结果。

这三个问题的共同根因是：**prompt 和 CLAUDE.md 里的规则，最终都只是 context 里的 token**。每一个 token 都要跟其他 token 竞争注意力。context 一拥挤，早期的硬约束就会被新进来的"我再试一下"挤到边缘。

靠写更好的 prompt 解决这个，等于用更多 token 跟 token 打架。方向错了。

**正确的方向是：把可以不放进 prompt 的东西拉出去，放在模型看不见的地方做确定性约束。** 这就是 harness 要解决的事。

## 一、五层全景

打个类比：Claude 像一个能力超强、学得飞快、但不守规矩的新同事。单靠口头说规矩做小任务没问题，但做严肃项目就必须有**门禁、团队百科、任务管理、同事反馈、SOP 手册**围着他。我搭的 harness 就是这五件事的自动化版本：

| 层 | 类比角色 | 解决什么问题 | 核心机制 |
| --- | --- | --- | --- |
| 1. 边界层 | 门禁 + 审计 | 硬拦截危险动作、验证证据 | hooks（工具调用前后自动跑的脚本） |
| 2. 记忆层 | 团队百科 | 几个月前的决策怎么找到 | 语义记忆 + 代码图谱 + 字面量搜索 + 时间轴查询 |
| 3. 交接层 | 任务管理 | 新 session 怎么接上昨天 | 双轨任务状态（人读协议 + 机器状态） |
| 4. 认知层 | 同事反馈 | 卡住了怎么跳出思维定势 | 7 种推理模式 + 自动切换 |
| 5. 技能层 | SOP 手册 | 高频工作流怎么固化 | 自定义 slash 命令 |

**这五层不是从架构书里抄的，是被坑出来的。** 每一层都对应一类具体事故。

## 二、边界层：把规则变成物理拦截

有一次我让 Claude 修一个 migration 报错。它试了四五种方案都不行，最后一步"为了清洁一下状态"，生成了一条会清空生产数据库全部数据、不可恢复的命令。

我当时已经在 CLAUDE.md 里加粗大字写了**"绝不 reset 生产数据库"**。Claude 读过、同意了、还是生成了它。

这件事让我学到一个硬道理：**规则写在 markdown 里是宪法，宪法靠合作。真正的边界必须是物理的。**

于是有了 **hooks**：每次 Claude 调用工具（执行 bash、编辑文件、停止对话）的前后，系统自动跑一段你指定的脚本，脚本可以拒绝调用、改写输入、注入上下文。**hooks 跑在 Claude 的 context 之外，Claude 看不见。**

我的第一个 hook 是 15 行 shell：每次 Claude 想执行 bash 前先扫一遍命令，命中 `DROP TABLE` / `TRUNCATE` / `rm -rf /` / `db reset` 就 exit 2 拒绝。Claude 收到的反馈是"这个命令没执行"，它无法合理化成"好吧我试别的"——因为它根本不知道为什么没执行，连拒绝原因都看不到。这一条 hook 救过我三次。

另一个关键用法是**验证闭环**。"不验证不提交"这条规则写在 markdown 里时，守约率大概七成；有三成它会说"已通过 tsc 检查"然后直接提交，其实根本没跑。我改成两个 hook 配合：编辑 `.ts` 后自动跑 `tsc` 和测试、跑完留一个带时间戳的"证据文件"；结束会话时再查——有改动但 10 分钟内没有证据文件？拒绝结束。精髓是：**不让模型自己声称"通过了"，让另一个脚本核实。** "我跑过 tsc"这句话必须有一个模型没法伪造的物理证据。

三个月下来，边界层的哲学压成一句：

> **CLAUDE.md 写承诺，hooks 写执行。承诺可以被合理化，执行不可以。**

## 三、记忆层：四工具分工，反直觉的是主力是 grep

高频痛点："这个决策三个月前做过了，理由我忘了——你还记得吗？"Claude 摇头，白纸。

主流做法是 RAG：把代码库和文档 embed 成向量，提问时按相似度召回 top-K。我花两周搭了一套完整 RAG，7000 多个 chunks、向量召回、时间衰减、三层缓存。跑三个月后统计：**日常 query 里语义 RAG 召回约三成；grep 反而是主力，占了一半多。**

不是 RAG 失败了。是我设计时就决定：**每种"查信息"的工具都有算法物理上不可替代的场景，强行用一个工具覆盖所有场景是性能浪费加正确性损失。** 举一个真实 query：*"为什么打卡和任务工时用了两张表？报表该选哪张？"* 这一句里夹了三种需求：

- **精确字面量**（字段名 `work_entries`）→ **grep 秒杀**，向量会把它模糊成"任何跟表有关的文档"；
- **中英术语 gap**（业务叫打卡、代码叫 attendance）→ **向量召回**，grep 在两者间 0 hits；
- **三个月前的决策 why** → **向量召回** 命中 decisions 文档。

再加两类：调用链/import 关系 → **代码图谱（AST graph）**；谁何时改过 → **git log/blame**。四个工具，四个不可替代的场景，**覆盖域 0 重叠**。

这里有个反主流的决策：**我拒绝装一个"智能路由器"去自动判断 query 该用哪个工具。** 理由很简单——模型在进化，今天 Claude 分不清的，明年的 Claude 能分清。我只在 prompt 里说清四个工具的边界，让它自己选。装路由器是把选择权从模型手里抢回来硬编码，锁死未来。

> **设计原则：把规则告诉模型，让模型选；不要做模型的代打。**

## 四、交接层：新 session 怎么接上昨天

把聊天历史塞进新 session 有两个致命问题：它是**高噪音流水账**（大部分是无效尝试、错误路径、回退），塞进去污染 context；它还会**超长**，自动压缩时关键结论会和噪音一起被压掉。

我的解法是**双轨任务状态**：一份人类可读的**任务协议**（YAML：当前功能、完成了哪些步骤、下一步、卡在哪），一份机器写的**运行时状态**（试过哪些方案、累计失败几次）。协议是*意图*，状态是*现场*。每次 `/start` 新 session，Claude 自动读协议 + 读运行时状态 + 查最近 3 天 session 日志 + 召回相关历史决策，编织成一张 **briefing**。这张 briefing 比 1M token 聊天历史质量高一个数量级，因为它是**结构化的结论**而非**过程流水账**。

这里有整个交接层**最锐利的一条边界**：任务协议和运行时状态**绝对不能进语义记忆索引**。任务协议是每 session 重写的当前态，半衰期是"本 session 结束"；一旦被 embed 进长期记忆，明年某次召回可能把今天的 TODO 注入到一个不相关的任务里制造幻觉。对比 session 日志——它**应该**进记忆，因为它是历史决策的载体。**区分"任务态"和"历史态"，同一个文件夹里不同文件有完全不同的生命周期**，决定了整个记忆系统会不会被污染。

## 五、认知层：卡住不是情绪，是信号

先说一件我**拒绝做**的事：PUA Claude。社区里有个流派用"你只剩 3.25 学分""你被解雇了"这种话术倒逼模型。我试过一周，结论是**技术上无效**（不是道德问题）：话术让模型做**角色扮演**，演一个被骂的员工，演戏消耗的是 prompt 能力不是解决问题的能力。**角色变了，推理范式没变。**

我吸收的抽象是另一个方向：**挫败感不是情绪，是信息熵信号。** 用户说"再试试""不对""又错了"——这些话**不带新约束**。没有新信息进入对话 = 上一轮方案没形成有效反馈 = **当前推理范式失败的确定信号**。正确的干预不是施压，是**换推理框架**。

于是给 Claude 配了 7 种推理模式（不是 7 种人设，是 7 种推理框架）：

| 模式 | 适用场景 | 核心方法 |
| --- | --- | --- |
| 根因分析 | 修 bug、查事故 | 5-Why + 扫同类 bug |
| 第一性原理 | 干净新建功能 | 质疑→删除→简化→加速→自动化 |
| 减法 | 重构、清理 | 删除优先，不增加新抽象 |
| 搜索优先 | 根因未知 | 先查历史/文档，再判断 |
| Working Backwards | 新模块设计 | 倒推用户终态 + 写 PR 稿 |
| 证据驱动 | 性能/质量测量 | 用数据替代直觉 |
| 闭环默认 | 部署/运维 | 定目标、追过程、拿结果 |

关键机制是**自动切换**：一个 hook 监听挫败感关键词，命中就注入一段**技术决策文本**（不是话术）——"标记当前方案为已失败，从 7 种模式重选一个，切之前先说明原模式失败在哪"。跟 PUA 的根本区别在于**失效边界的诚实**：用户自己问错了问题时，我这套会**静默降级**不制造压力；PUA 在所有场景等价施压，分不清是"模型思路错了"还是"用户问错了"。**Agent 治理的核心不是让模型听话，是给模型一个能跳出思维定势的机制。**

## 六、技能层：把工作流固化成 SOP

**skill = 一个 slash 命令 + 一份 markdown 文件**，告诉 Claude"用户敲这个命令时，按这个顺序做这些事"。不写代码、不配 MCP、不装插件。我用得最多的是 `/start`（开场 briefing）、`/commit`（按文件类型路由验证，全绿才生成 commit）、`/handoff`（收尾仪式）。

`/handoff` 最能展示 skill 设计的威力——**一个命令触发跨层协同**：更新任务协议（交接层）、写 session 日志、flush 记忆 pending 队列 embed 进长期记忆（记忆层）、把失败计数和当前推理模式原子持久化（认知层），而其中的 Edit 又会触发编辑后 hook（边界层）。用户敲一个"收工"，背后是整个 harness 在做收尾。

这份 markdown 里有几个我踩坑学到的设计：**description 里写"触发词"**（"今天到这"字面跟 handoff 无关但语义上是，让模型做意图匹配）；**性能约束写进 skill**（"≤ 4 tool calls"，否则它会"认真"地挨个调用变成 12 次）；**并行调用明确标注**（独立的 git log 和 read 在同一条消息里发，latency 从线性降到常数）；**Fast/Full 分级**（日常收尾跑 4 次调用，大 sprint 才 `--full` 跑 8 个固化判定）；**条件触发**（没有值得固化的 pattern 就不写、避免污染）。

skill 是 Unix 哲学在 LLM 时代的转译：do one thing well、small tools、compose via pipes——唯一区别是**脚本是 markdown 写的，执行者是 LLM 而不是 shell**。markdown 写错改一行成本接近零，代码写错要重构成本高一个数量级。我现在项目里 20 多个 skill、加起来不到 5000 行 markdown，换来每天敲几百次命令的确定性——**这是整个 harness 投入产出比最高的一层**。

## 七、转折：五层都架在别人的地基上

五层做完，我撞到一堵墙。

这五层——hooks、记忆、交接、认知、技能——**全部架在 Claude Code 暴露的那几个扩展点上**。它给你一个封闭 runtime，开几个口子，你在口子上打补丁。一开始有效。直到它一次更新内部数据结构变了，我的 hooks 集体失效；直到任务复杂到单轮装不下，我的调度器只能用 hack 模拟多轮；直到我想做"冻结前缀"这种精确的 token 控制时，发现它根本不暴露这个接口。

有三件事，我无论怎么写 prompt、怎么挂 hook 都够不到：

- 我不能重写**主循环**——它定死了"调工具 → 等结果 → 继续"，我无法让它在一步失败后**换一种推理范式**回退到目标解构再重来；
- 我不能定义**编排拓扑**——模型支持几百路并发，可对话式调度器只会单线程串行调工具，因为它从设计上就是为"聊天"而非"并行编排"造的；
- 我不能控制**可靠性的天花板**——上下文何时截断、失败怎么重试，全写死在别人的实现里，API 里压根没有"不要截断"这个选项。

这不是"灵活性不够"，是**控制权的结构性缺失**。每一层补丁都在为同一件事买单：**底层不属于我。**

转折发生在一个深夜。我想明白：我要的从来不是"在某个工具上跑通 demo"，而是**我的工程判断能被 agent 忠实执行**——而品味不是靠软约束注入的，它必须刻进底层硬结构里。

于是我找到了 **pi**：一个精简到骨头的开源 agent runtime，只跑主循环、暴露接口，不替你做任何决定。我 fork 了它，把那五层从"补丁"重铸成"内建结构"，再往下叠了三层 userland 永远够不到的东西：确定性编排、跨模型校验、会自己整理的记忆。

这就是 **羲和 Xihe**——神话里执掌十日、把太阳一颗颗送上天的太阳之母。它不是一个 agent，是 agent 立足的那片天空和大地。**可靠性不再求着模型自觉，而是从模型之外，被一块块工程出来。**

---

## 八、拥有 harness 之后：模型之外，能焊得更深

这一层是分水岭：因为可靠性活在**我自己的代码**里，我现在敢用 DeepSeek flash、MiMo 这种**又小又便宜的模型**去跑生产活——harness 焊的护栏，让它表现得像个有纪律的工程师。下面几道闸，都是闭源 harness 上做不出来的。

**① 确定性编排：把意图切成 DAG。** 一个 conductor 把任务分解成一张 DAG，每片叶子是一个原子产物、一条独立故障边界，带着它在"完成"之前必须断言的后置条件。独立叶子**并发跑**，共享的上下文用冻结前缀只算一次、被每片叶子继承——这让宽并发付得起。角色彻底解耦：conductor 规划、executor 执行、verifier 校验，每个角色绑哪个模型是配置里拧的旋钮，不是写死的。

**② 跨模型校验器 + 静默升级：不要静默的错答案。** 弱模型最隐蔽的坑是选了条很弱的路径还**静默交付**，看着煞有介事其实是错的。羲和让一个**专门唱反调的怀疑者**在 DAG 跑完后逐条对照任务要求审结果——而且**故意用跟干活的模型不同的模型**去审。为什么？*同模型自审会复用同一盲点，它造的坏计划自己看不出坏*。校验判 fail、且你配了更强的模型时，conductor **静默升级**：换个更强的脑子、带着失败原因重新规划重跑。没配 SOTA key？优雅降级，继续用便宜模型，并把"没升级"明确记下来，绝不静默丢护栏。经济学是关键——conductor 每个任务只规划一次，但一张坏图会让**一整轮叶子白干**，风险不对称，所以正确姿势是：**弱模型造图，校验器兜底，失败才升级。**

**③ 结构化输出硬闸：别信，校验，失败就 fail-closed。** 弱模型爱返"几乎对"的 JSON——漏字段，或最经典的：把 JSON 裹进一个 markdown 代码块再吐给你，下游 `JSON.parse` 直接炸（我跑过一组评测，某弱模型在"纯自然语言调工具"上三次全挂，全死在这）。最偷懒的修法是"换强模型"，我没这么干。羲和的修复闸是确定性的：先剥 fence、提取 JSON → schema 校验 → 没过就把具体错误喂回**同一个模型**重问一次 → 还不过就 **fail-closed**，明确报错，**绝不返一个半对的对象去骗下游**。同模型、只重问一次、失败就硬失败。*弱模型的 harness 是焊护栏，不是堆算力。*

**④ 法定数字 grounding：幻觉不分模型强弱。** 模型会很自信地给你一个数字（某个税率、某条法规的生效日），而它的记忆停在了某个旧版本——这事 Opus 也会犯，是 model-agnostic 的。所以它不该靠"给弱模型多写一句提示"治，该靠一道**跟模型无关的代码闸**：扫描输出、命中"必须有出处"的声明就检查有没有引文，没有就按严格度追加免责声明或直接拦。*软提示我也保留，但那只是软半边——软标签本身不足信，真护栏在硬闸。*

**⑤ Q-router：该用哪个模型，别猜，让它学。** "这类活该用哪个模型"凭什么我拍脑袋写死？羲和挂了一个 ε-greedy bandit，对每类执行单元从你配置的候选池里选，跑完按真实回报更新——**回报信号就是上面那个 verifier 过没过**。两个让我敢 ship 进开源默认的设计：冷启动等同今天（候选池第一个就是你原来的静态默认，optimistic 初始化保证先各试一次，**第一天行为和没有它时一模一样**），以及零配置零回归（池子只有一个模型就整体 no-op）。它不 ship 任何预训练权重，学的是**你的**工作负载。

**把它们串成一条命令。** 我每天派活用一个脚本叫 `xihe-build`，给它一句话目标加一份契约，它会：conductor 把目标拆成 DAG → 执行叶子用 **flash（那个又便宜又不算聪明的模型）真改文件**（行锚定编辑专治弱模型那种"行错位"腐烂、LSP 实时喂回语法诊断）→ 在 DAG 里插一个 `tsc` 节点中途自检 → 整张图跑完过一道**确定性 oracle 闸**（`tsc`，带参数时加 `bun test`）——不是问模型"你觉得对吗"，是编译器说了算 → 红了不盲目重试，一个分类器先判这是"可修复"还是"编译器级硬错"还是"自愈几轮零进展卡住了"，硬错和卡住直接提前止损，可修复才派一个 fix-DAG 喂错误摘要重修、有界轮数 → **只有绿了才触发对抗式 review**（审红代码只会让 findings 抖动），而且风险门控：动了关键区（账本/schema 这类）自动升到更严的档去抓"契约违反" → 全程记进自进化飞轮。

从头到尾，**那个弱模型敲下了每一个字符。但没有一行可靠性活在它的 prompt 里。** 可靠性活在 `tsc` 闸里、活在自愈循环的分类器里、活在 review 门控里、活在"工作树不干净就拒跑""跑完不自动 commit、等我审 diff"这些规矩里。

## 九、双线落点：开源底座，真实项目自证

两半到这儿合上：

- **底座开源，品味归你。** 开源出来的是通用 harness——编排、记忆、技能、护栏。你的业务逻辑、品牌、密钥仍是你的；你可以往记忆里注入自己领域的"领域包"，让答案带可核查的来源，而开源底座绝不吸收你的私货。羲和已经在 [github.com/AbyssCN/xihe](https://github.com/AbyssCN/xihe)。
- **真实项目自证。** 这套 harness 不是 demo——上面那五层 + 三层，现在跑在一个真实生产项目里（就是开头那个高约束 B 端 SaaS），每天用 `xihe-build` 这类脚本派活、自愈到绿、门控审查。它经得起"错一个 migration 就停机"的场景，才敢说它可靠。

模型是引擎，harness 是底盘。给它任何 LLM——DeepSeek、MiMo、Qwen、Claude——可靠性来自这套系统，所以它跟着**你**走，不跟着模型走。

## 结尾：回到"本质"

Harness Engineering 的本质是什么？五层加三层做下来，底层哲学很干净：

> **可靠的系统承载不可靠的模型。**

模型是熵源——输出有概率、会 hallucination、会被上下文污染、有被"说服跳过规则"的空间。系统的确定性必须来自**模型之外**：hook 的拒绝、记忆层那几十行决定什么进什么不进的 regex、人手写的任务协议、按关键词注入的推理模式切换、`tsc` 的 oracle 闸、校验器的怀疑——这些都不经过模型，模型也改不动。那模型在哪里参与？**在每一层的中心位置**——决定用哪个工具、召回什么、用哪种推理模式、每一步的具体参数。**模型做决策，系统保正确。**

这跟主流方向是反的。主流相信"模型越强越好，能塞进 prompt 就塞进 prompt"。我的判断反过来：

**三个 takeaway：**

1. **vibe coding 在复杂项目是物理上失效的。** 不是 prompt 技巧问题，是长上下文衰减、跨 session 失忆、规则一致性这三件事靠提示词无法解决。复杂项目必须有 harness。
2. **别跟模型讲道理，给模型写 kernel。** CLAUDE.md 是协议、靠合作；真正守边界的是跑在模型看不见维度上的代码闸——模型没法合理化它不知道存在的约束。
3. **"模型够强就够了"是错的，反过来才对：模型越弱、越便宜，harness 越是可靠性本身。** 因为可靠性一旦写在模型之外的代码里，它就是 model-agnostic 的——这才让一个 DeepSeek flash 敢去跑你平时只敢交给前沿模型的活。而当你想把这套挽具焊到最深处时，你会发现：你得**拥有**那个 runtime，而不是在别人的地基上贴便条。

不是学更好的 prompt。是搭一套能让 prompt 不那么重要的系统——并且，让那套系统归你。
